# DVS Specification Skill

## Purpose

Convert validated XLSForm files (from the edc-builder skill) into a
fully populated OpenClinica 4 Data Validation Specification (DVS) workbook,
and write approved changes from an edited DVS back into the source XLSForms.

**Template file:** `references/DVS_Template.xlsx`
Always copy this template as the basis for every output — never build the
workbook from scratch. The template carries Lookups, OC4_Syntax_Guide,
Examples, and README sheets that must be preserved unchanged.

---

## Before You Begin — Read Reference File

**Always read `references/dvs-column-mapping.md` before processing any
input.** It documents:
- Exact DVS_OC4 column definitions and valid values
- XLSForm-to-DVS field mapping rules
- Check derivation rules per XLSForm row type
- Write-back rules (DVS → XLSForm)
- ID numbering conventions

---

## Input Detection — Two Modes

Inspect what was provided before doing anything else.

### Mode A — XLSForms → Generate DVS

**Trigger:** User provides an EDC Builder zip package, a directory of
XLSForm `.xlsx` files, or individual XLSForm files. No DVS xlsx provided.

**Identify:** Files have `settings` / `choices` / `survey` sheets and
no `DVS_OC4` sheet.

→ Follow **Steps 1–6** (Generate flow).

### Mode B — Updated DVS → Write Back to XLSForms

**Trigger:** User provides a DVS xlsx that was previously generated by
this skill (identified by the presence of a `DVS_OC4` sheet) alongside
XLSForm files or a build zip.

**Identify:** File has a `DVS_OC4` sheet.

→ Follow **Steps 7–9** (Write-back flow).

## Step 1: Read the XLSForm Inputs

### 1a: Locate XLSForm Files

If a zip is provided, extract it and locate all `.xlsx` files inside
the `forms/` directory. If individual files are provided, use those
directly.

For each XLSForm, read all three sheets:
- `settings` → form_title, form_id, crossform_references, namespaces
- `choices` → all list_name / label / name rows
- `survey` → all rows; the header row defines column positions

### 1b: Extract Study Metadata

From any form's settings sheet:
- `form_id` list → full list of forms in the study
- Study ID (inferred from `pulldata('{study_id}_tpt',...)` in any
  TPTCALC calculation cell if present)
- Protocol number (from zip filename or user context)

### 1c: Build Form and Item Index

For every form, build an internal index:
```
{
  "form_id": "AE",
  "form_title": "Adverse Events",
  "crossform_references": "SE_SCREEN",
  "items": [
    {
      "name": "AETERM",
      "label": "Adverse event term:",
      "type": "text",
      "constraint": "",
      "constraint_message": "",
      "required": "yes",
      "relevant": "${AEYN}='Y'",
      "calculation": "",
      "readonly": "",
      "bind_oc_external": "clinicaldata",   # or blank
      "bind_oc_constraint_type": "hard",    # or blank
      "choice_filter": "",
      "itemgroup": "AE"
    }
  ]
}
```

## Step 2: Derive DVS Checks from XLSForm Rows

For every survey row in every form, apply the derivation rules below.
Each derivable row becomes one row in DVS_OC4 and one row in
Protocol_Extraction. Skip rows where the relevant column is empty AND
constraint is empty AND required is blank AND the row is a structural
type (begin/end group/repeat, calculate without clinicaldata external).

**Check ID numbering:** `DVS-001`, `DVS-002`, ... sequentially across
all forms, ordered by form then by row position within form.

**Query Text ID numbering:** `QT-001`, `QT-002`, ... sequentially,
one per unique message string.

**UAT Case ID numbering:** `UAT-001`, `UAT-002`, ... one per DVS check.

### 2a: Constraint Rows

**Trigger:** row has a non-empty `constraint` value.

| DVS_OC4 field | Derived value |
|---|---|
| Check Type | `Constraint` |
| Severity | `Hard` if `bind::oc:constraint-type = hard`; else `Soft` |
| Trigger Point | `Real-time on form entry` |
| Expression / Calculation | The constraint expression verbatim |
| Constraint / Required / Relevant Message | constraint_message value |
| Target Form OID | form_id |
| Target Item Name | label (cleaned — strip `**`, trailing `:`) |
| Target Item OID | name |
| OC4 Logic Pattern | `Same-form constraint` if no cross-form refs; else `Cross-form helper + constraint` |
| Helper Calculate Item Needed? | `Yes` if calculation references `instance('clinicaldata')`; else `No` |

### 2b: Required Rows

**Trigger:** row has `required = yes` or a non-trivial required expression
(not just `yes` or `true()`). Skip `required = yes` on rows that also
have a constraint — the constraint check already covers the row; add a
note instead.

| DVS_OC4 field | Derived value |
|---|---|
| Check Type | `Required` |
| Severity | `Hard` if `bind::oc:required-type = hard`; else `Soft` |
| Trigger Point | `On item completion` |
| Expression / Calculation | required expression (blank if simply `yes`) |
| OC4 Logic Pattern | `Required field` |

### 2c: Relevant / Show-Hide Rows

**Trigger:** row has a non-empty `relevant` expression AND row type is
a data entry type (not begin/end group — groups with relevant are
structural, not a standalone check). Generate one check per form
section if a group's `relevant` is the controlling logic.

| DVS_OC4 field | Derived value |
|---|---|
| Check Type | `Relevant` |
| Severity | `Informational` |
| Trigger Point | `Real-time on form entry` |
| Expression / Calculation | relevant expression verbatim |
| OC4 Logic Pattern | `Conditional display` |

### 2d: Cross-Form Helper Calculate Rows

**Trigger:** row type is `calculate` AND `bind::oc:external = clinicaldata`.

| DVS_OC4 field | Derived value |
|---|---|
| Check Type | `Cross-Form Helper` |
| Severity | `N/A` |
| Trigger Point | `Real-time on form entry` |
| Expression / Calculation | calculation verbatim |
| OC4 Logic Pattern | `Cross-form helper calculate` |
| crossform_references | the event OID(s) referenced in the XPath |
| Current Event Needed? | `Yes` if `@OpenClinica:Current='Yes'` in XPath; else `No` |
| Helper Item OID | name of this calculate field |
| Helper Calculate Item Needed? | `Yes` |
| Source Event OID(s) | extract from XPath |
| Source Form OID(s) | extract from XPath |
| Source Item OID(s) | extract from XPath |

### 2e: Derivation / Calculation Rows

**Trigger:** row type is `calculate` AND `bind::oc:external` is blank
AND calculation is not a simple pulldata() call.

| DVS_OC4 field | Derived value |
|---|---|
| Check Type | `Derivation / Review Listing` |
| Severity | `N/A` |
| Trigger Point | `Real-time on form entry` |
| Expression / Calculation | calculation verbatim |
| OC4 Logic Pattern | `Calculated derivation` |

### 2f: Check Scope Classification

For each check, determine `Event Scope` and `Source Event OID(s)`:
- If constraint/expression references only fields within the same form:
  → `Event Scope = Single event`; Source Event OID = the form's event(s)
- If expression references `instance('clinicaldata')` with a specific
  event OID: → `Event Scope = Cross-event`; Source Event OID = the
  referenced event OID(s)
- If expression references `@OpenClinica:Current='Yes'`:
  → `Current Event Needed? = Yes`

## Step 3: Populate Protocol_Extraction

For every check derived in Step 2, write one row to Protocol_Extraction:

| Col | Value |
|-----|-------|
| Source Section | Map from context: constraint → field's form category; required → "Data Completeness"; relevant → "Conditional Logic"; cross-form → "Cross-Form Validation" |
| Protocol Reference | Leave blank (PLACEHOLDER — populate from protocol) |
| Category | Map: constraint/required → "Data Quality"; relevant → "Conditional Display"; cross-form → "Cross-Form Validation" |
| Structured Requirement / Fact | Generate from check name and expression in plain English |
| Raw Protocol Text Summary | Leave blank (PLACEHOLDER) |
| Downstream Build Object | `Item` for field-level checks; `Form` for form-level; `Group / Section` for group relevant |
| Potential Check Needed? | `Yes` |
| Candidate Check ID | The assigned DVS Check ID |
| Related Event OID | Source Event OID(s) |
| Related Form OID | Target Form OID |
| Related Item Name / OID | `{Target Item Name} / {Target Item OID}` |
| Priority | `Critical` for Hard checks; `High` for Soft; `Medium` for Informational; `Low` for Derivation |
| Owner | Leave blank (PLACEHOLDER) |
| Status | `Draft` |
| Notes | Any flag reasons or context from the source XLSForm row |

## Step 4: Populate DVS_OC4

Write all columns for each check. Use the valid values from the
Lookups sheet for Check_Type, Severity, Trigger_Point, Status.

**Default Status for all generated rows:** `Draft`

For columns that cannot be derived from the XLSForm:
- `Business Purpose` → generate from check name and context
- `Protocol Reference` → `[PLACEHOLDER — add protocol section reference]`
- `Build Owner` → leave blank
- `Expected Site Action` → generate a plain-English instruction appropriate
  to the check type (e.g. "Verify age and correct if entered in error.")
- `Notes` → copy any flag_reason from source XLSForm row

---

## Step 5: Populate Query_Text_Library

For every check that has a `Constraint / Required / Relevant Message`:
1. Check whether the message string already appears in the library
   (de-duplicate identical messages)
2. If new: assign next QT-ID, write a row
3. Link the QT-ID back to the DVS_OC4 row's `Query Text ID` column

| Col | Value |
|-----|-------|
| Query Text ID | QT-001, QT-002, ... |
| Status | `Draft` |
| Standard Message | The constraint_message / required_message verbatim |
| Audience | `Site` for Hard/Soft; `Review` for Informational |
| When to Use | Generate from check type context |
| Avoid / Notes | `[PLACEHOLDER — add any wording guidance]` |
| Related Check ID(s) | Comma-separated list of DVS Check IDs using this message |
| Priority | Match DVS Priority |
| Owner | Leave blank |
| Version Notes | `Generated from XLSForm — verify wording` |

## Step 6: Populate UAT_Cases

For every DVS check, generate at least one UAT case. Generate two cases
for Hard checks (a pass scenario and a fail scenario).

**Columns 1–16 (human test spec):**

| Col | Value |
|-----|-------|
| UAT Case ID | UAT-001, UAT-002, ... |
| Status | `Not Run` |
| Related Check ID | DVS Check ID |
| Scenario | Generate: "[Pass/Fail] — [Check Name]" |
| Preconditions | Generate from form and event context |
| Test Steps | Generate plain-English steps: open form, enter data, attempt save |
| Input Data | Generate representative pass/fail data from the expression (e.g. for `. >= 18`: pass = `18`, fail = `17`) |
| Expected Result | Generate from check type and severity |
| Actual Result | Leave blank |
| Test Result | `Not Run` |
| Tester | Leave blank |
| Execution Date | Leave blank |
| Defect / Ticket | Leave blank |
| Retest Needed? | Leave blank |
| Priority | Match DVS Priority |
| Notes | Leave blank |

**Input data inference rules:**

| Expression pattern | Pass input | Fail input |
|---|---|---|
| `. >= N` | N | N-1 |
| `. <= N` | N | N+1 |
| `. >= N and . <= M` | N | N-1 or M+1 |
| `. <= today()` | today's date | future date |
| `. >= ${X}` | value equal to X | value before X |
| `. = 'Y'` | Y | N |
| `. = 'N'` | N | Y |
| `regex(., 'HH:MM pattern')` | `09:30` | `9:30` or `25:00` |
| `not(selected(.,'X') and count-selected(.)>1)` | single selection | X + another |

**Columns 17–25 (ODM load coordinates):**

Cols 17–18 (`Site_OID`, `Participant_Key`) are **always left blank** in the
DVS — they are stamped at runtime by `uat_loader.py`.

Cols 19–25 are populated by this skill from XLSForm metadata:

| Col | How to populate |
|-----|----------------|
| Study_Event_OID (19) | From `forms[].visits_assigned[0]` in Study Spec JSON, or `crossform_references` in XLSForm settings. Use the first event. If form is multi-event and item is a data field, write `[PLACEHOLDER — multi-event form, confirm target visit]`. Leave blank for calculate/readonly/structural rows. |
| Event_Repeat_Key (20) | Default `1`. Write `2`, `3`, ... only for UAT cases that explicitly test repeating event instances. Leave blank for calculate/readonly/structural rows. |
| Form_OID (21) | `form_id` from XLSForm settings sheet for the form containing this item. Leave blank for structural rows. |
| Item_Group_OID (22) | `bind::oc:itemgroup` value from the survey row. Leave blank for begin/end group, calculate, and note rows. |
| Participant_ID (23) | Default `UAT-P001`. Assign `UAT-P002` only when the Preconditions column explicitly requires a different baseline state (e.g. female participant). All rows in a cross-form dependency chain share the same Participant_ID. |
| Load_Order (24) | Integer sequence by form group: DM/ICF = 1–10, IE/MH = 11–20, VS/PE/LB = 21–50, AE/CM/EX = 51–80, DS/safety = 81–99. Within a form, rows load in survey row order. Cross-form source rows must have lower Load_Order than dependent rows. Leave blank for non-loadable rows. |
| Load_Value (25) | Pass-scenario value from Input Data (col 7), normalised: dates as ISO `YYYY-MM-DD`, select_one as choice `name` (not label), numbers as plain numeric strings. Leave blank for calculate, readonly, and fail-scenario rows. |

**Rows where all ODM columns (17–25) must be blank:**
- `begin group`, `end group`, `begin repeat`, `end repeat`
- `calculate` rows (any)
- `note` rows
- `readonly` rows
- `EVENT_CF` row

## Step 6a: Write UAT_Setup Tab

After populating UAT_Cases, write the `UAT_Setup` information tab.
This tab has **fixed content** — it does not vary by protocol.
No extraction or inference is needed; the `generate_dvs.py` script
writes it automatically from the `_UAT_SETUP_ROWS` constant.

When calling `build_dvs()`, you do not need to supply a
`calendaring_rules` or `calendaring_uat` key if no calendaring rules
were extracted — the script defaults to empty lists and still writes
the tab headers.

## Step 6b: Populate Calendaring_Rules

If the Study Spec JSON contains a `calendaring_rules` array (added by
the `protocol-analysis` skill, Step 8a), convert each entry to a
Calendaring_Rules row.

| DVS Column | Source |
|------------|--------|
| Rule ID | `calendaring_rules[i].rule_id` |
| Status | `Draft` |
| Rule Name | `rule_name` |
| Business Purpose | `description` |
| Protocol Reference | `protocol_reference` |
| Trigger Type | `trigger_type` |
| Trigger OID | `trigger_oid` (blank if null) |
| Schedule | `schedule` (blank if null) |
| Schedule Time | `schedule_time` (blank if null) |
| Condition (XPath) | `condition_xpath` (`$TRUE` if unconditional) |
| Condition (Plain English) | `condition_plain_english` |
| Action Type | `actions[0].action_type` (one row per action; repeat Rule ID for multi-action rules) |
| Target Event OID | `actions[0].target_event_oid` |
| Target Form OID | `actions[0].target_form_oid` |
| Action Parameters | `actions[0].parameters` serialised as compact JSON |
| Rule Result To Trigger On | `true` (default unless spec says false) |
| JSON Output | Assemble from all fields — see OC4_Syntax_Guide sheet for the full JSON schema |
| Build Owner | Leave blank |
| Priority | `priority` from spec (default `High`) |
| UAT Case ID(s) | Leave blank — linked in Step 6c |

If no `calendaring_rules` key is present in dvs_data (e.g. because the
Study Spec was generated before the protocol-analysis skill was updated),
write an empty Calendaring_Rules sheet with headers only and a single
placeholder row noting `[No calendaring rules extracted — populate manually]`.

## Step 6c: Populate Calendaring_UAT

For each row in Calendaring_Rules, generate one `CUAT-NNN` test case.

| Col | Value |
|-----|-------|
| UAT Case ID | CUAT-001, CUAT-002, ... |
| Status | `Not Run` |
| Related Rule ID | CAL-NNN |
| Scenario | Generate from Rule Name (e.g. "Verify: [Rule Name] fires correctly") |
| Preconditions | Generate from trigger type and condition (e.g. "UAT participant with DM data loaded. Baseline event in Data Entry Started status.") |
| Setup Steps | Generate numbered steps to reach the precondition state using the dated UAT site from the most recent data load run |
| Trigger Action | Generate from Trigger Type (e.g. "Click Close on the [form] form for the [event] event") |
| Expected Outcome | Generate from Action Type and parameters (e.g. "Visit 2 event appears in participant timeline scheduled 28 days after Baseline start date") |
| Verification Steps | Generate numbered steps to confirm the outcome in OC4 UI |
| Actual Result | Leave blank |
| Test Result | `Not Run` |
| Tester | Leave blank |
| Execution Date | Leave blank |
| Defect / Ticket | Leave blank |
| Notes | Leave blank |

After writing Calendaring_UAT, go back and fill `UAT Case ID(s)` in the
corresponding Calendaring_Rules rows.

## Step 7 (Mode B): Read the Updated DVS

When a DVS xlsx is provided for write-back:

### 7a: Identify Changed Rows

Read DVS_OC4. For each row, compare against the previously generated
state by matching on Check ID. A row is considered changed if any of
these columns differ from what was built into the XLSForm:
- `Expression / Calculation`
- `Constraint / Required / Relevant Message`
- `Severity` (Hard vs Soft — affects bind::oc:constraint-type)
- `Check Type` (change in which XLSForm column the expression lives)
- `Status` — if changed to `Retired`, the expression should be cleared
  from the XLSForm

### 7b: Validate Changes Before Write-Back

Before applying any change, validate:
- Target Form OID exists in the provided XLSForm files
- Target Item OID (name) exists in that form's survey sheet
- Expression is valid XLSForm syntax (does not contain unmatched
  brackets or obviously broken XPath)
- If validation fails, log to write-back report and skip that row

---

## Step 8 (Mode B): Apply Changes to XLSForms

For each validated changed row, locate the target survey row in the
target XLSForm (match on `name` column = Target Item OID) and update:

| DVS change | XLSForm column updated |
|---|---|
| Expression changed AND Check Type = Constraint | `constraint` |
| Expression changed AND Check Type = Required | `required` |
| Expression changed AND Check Type = Relevant | `relevant` |
| Expression changed AND Check Type = Derivation | `calculation` |
| Expression changed AND Check Type = Cross-Form Helper | `calculation` |
| Message changed | `constraint_message` or `required_message` |
| Severity changed to Hard | Add `bind::oc:constraint-type = hard` |
| Severity changed from Hard | Clear `bind::oc:constraint-type` |
| Status = Retired | Clear the expression column; clear message |

**Never update:** `type`, `name`, `label`, `bind::oc:itemgroup`,
`bind::oc:external`, `appearance`, `readonly`. These are structural
and outside DVS scope.

---

## Step 9 (Mode B): Package Write-Back Outputs

1. Save each modified XLSForm back to the forms directory
2. Re-zip the full package: same structure as the original build zip
3. Generate a **Write-Back Report** (text or small XLSX) listing:
   - Every field updated (form, item name, column, old value, new value)
   - Every row skipped with reason
   - Count of forms modified
4. Present the new zip and write-back report to the user

## Step 10: Generate DVS Output Files

Produce **two output files** on every Mode A run: the XLSX workbook
(the complete working document) and the PDF (the curated audit artifact).

### XLSX

```python
import sys
sys.path.insert(0, '/mnt/skills/user/dvs-specification/scripts')
from generate_dvs import build_dvs

xlsx_path = '/mnt/user-data/outputs/{PROTOCOL_NUMBER}_DVS.xlsx'
build_dvs(dvs_data, xlsx_path)
```

The XLSX script:
1. Opens the template at `references/DVS_Template.xlsx`
2. Preserves README, Lookups, OC4_Syntax_Guide, Examples unchanged
3. Writes Protocol_Extraction, DVS_OC4, Query_Text_Library
4. Writes UAT_Cases (25 columns — 16 test spec + 9 ODM load coordinates)
5. Writes UAT_Setup (fixed info tab — no dvs_data input required)
6. Writes Calendaring_Rules (from `dvs_data['calendaring_rules']`; empty headers if key absent)
7. Writes Calendaring_UAT (from `dvs_data['calendaring_uat']`; empty headers if key absent)
8. Enforces sheet order: README → Lookups → Protocol_Extraction → DVS_OC4 →
   Query_Text_Library → UAT_Cases → UAT_Setup → Calendaring_Rules →
   Calendaring_UAT → OC4_Syntax_Guide → Examples
9. Applies consistent formatting; PLACEHOLDER cells highlighted amber
10. Saves as `{PROTOCOL_NUMBER}_DVS.xlsx`

### PDF

```python
from generate_dvs_pdf import build_dvs_pdf

pdf_path = '/mnt/user-data/outputs/{PROTOCOL_NUMBER}_DVS.pdf'
build_dvs_pdf(dvs_data, pdf_path)
```

The PDF is a **sign-off and audit artifact** — curated, not a full column
dump. It contains:

| Section | Contents |
|---------|----------|
| Cover | Protocol number, study ID, generated date, review status |
| Section 1 — DVS Summary | Check counts by type / severity / priority / status; UAT and calendaring totals |
| Section 2 — Protocol Extraction | Source requirements traceability table (Category, Requirement, Protocol Ref, Check ID, Priority) |
| Section 3 — DVS Checks | Curated 9-column view: Check ID, Check Name, Type, Severity, Target Form, Target Item, Expression, Expected Site Action, Status. Row shading: Red = Hard, Amber = Soft, Green = Informational |
| Section 4 — Query Text Library | QT ID, Standard Message, Audience, Related Check IDs, Status |
| Section 5 — UAT Summary | Pass/Fail/Not Run counts, pass rate, case list with result shading |
| Section 6 — Calendaring Rules Summary | Rule ID, Name, Trigger Type, Condition, Action Type, Priority, Status |
| Section 7 — Approval Block | Signature table for Data Manager, CPM, Sponsor Representative |

The PDF omits the ODM load coordinate columns (UAT_Cases cols 17–25),
write-back expressions, full XPath detail, and the UAT_Setup tab — these
belong in the XLSX working document.

Present both files to the user after generation.

## Human Review Instructions

At the end of every Mode A run, include this block:

```
─────────────────────────────────────────────────────
DVS REVIEW REQUIRED — DO NOT MARK APPROVED UNTIL COMPLETE
─────────────────────────────────────────────────────
Before marking any check as Approved:

1. COMPLETE all [PLACEHOLDER] cells in DVS_OC4 and Protocol_Extraction
   — especially Protocol Reference and Business Purpose

2. VERIFY all generated Expression / Calculation values are correct
   for the intended validation logic

3. CONFIRM Severity (Hard vs Soft) for every constraint check

4. REVIEW all Query Text messages — wording must be plain-language
   and suitable for site data entry staff

5. REVIEW UAT_Cases — confirm Pass/Fail input data is representative,
   Expected Results are accurate, and Load_Value (col 25) contains
   the correct machine-readable value for each pass-scenario row

6. REVIEW UAT_Setup tab — fixed content; confirm it accurately
   describes your environment (no edits normally needed)

7. REVIEW Calendaring_Rules — confirm Trigger Type, Condition (XPath),
   Action Parameters, and JSON Output for every rule; the JSON Output
   column is what gets pasted into OC4 Rules Management

8. REVIEW Calendaring_UAT — confirm Setup Steps and Verification Steps
   reference the correct events, forms, and expected outcomes

9. CHANGE Status from Draft → In Review → Approved as review progresses

10. When ready to write approved changes back to XLSForms, provide
    this DVS xlsx alongside the XLSForm files or build zip.
─────────────────────────────────────────────────────
```
