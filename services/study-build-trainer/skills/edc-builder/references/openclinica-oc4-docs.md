# OpenClinica 4 Documentation — Reference Guide

**Authoritative source:** https://docs.openclinica.com/oc4/

This file is a **curated index** into the OpenClinica 4 user documentation,
organized by relevance to the EDC Builder skill. Do NOT rely on cached
snapshots of this content — always fetch the live page when you need
current guidance, because OpenClinica updates docs between releases.

---

## Priority sections for form/XLSForm building

These are the pages most directly relevant to generating XLSForm `.xlsx`
files that will import cleanly into OpenClinica 4:

### 2. Building Forms and Studies
**Landing:** https://docs.openclinica.com/oc4/building-forms-and-studies/

#### 2.3 Form Library (CDASH)
https://docs.openclinica.com/oc4/building-forms-and-studies/cdash-crf-library/
→ CDASH-standard forms available in the OpenClinica Form Library.
When form_category is CDASH_CLINICAL or CDASH_SAFETY, field names,
item group conventions, and choice lists should align with CDASH
naming here.

#### 2.4 Design a Study
**Landing:** https://docs.openclinica.com/oc4/building-forms-and-studies/oc4-design-study/

Subsections (anchor links from the landing page):
- **2.4.1 Events & Forms** (`#content-9133`)
  Study Designer event structure — how events, visits, and form cards
  are organized. Relevant for `timepoint_csv` + form `visits_assigned`.
- **2.4.2 Using Form Designer** (`#content-9184`)
  The web-based Form Designer that consumes XLSForm uploads. Errors
  encountered during upload typically trace back to violations of
  rules documented here.
- **2.4.3 Designing Contact Data Forms** (`#content-32533`)
  Only applicable to forms using the `instance::oc:contactdata`
  external binding — usually just F_CONTACT or equivalent.
- **2.4.4 Using the Form Template** (`#content-9204`)
  **Authoritative source for XLSForm template structure.** All
  column names, required/optional fields, and sheet ordering rules
  originate here. Cross-reference against
  `references/xlsform-build-rules.md` when resolving ambiguity.
- **2.4.5 Form Logic** (`#content-17316`)
  XPath expressions for `relevant`, `constraint`, `calculation`.
  Includes OC-specific extensions beyond standard XLSForm XPath.
- **2.4.6 Functions** (`#content-18363`)
  Supported functions in expressions. The Validated Functions Index
  (2.4.6.1 at `#content-22914`) lists every function acceptable in
  OpenClinica forms — stick to this list to avoid unsupported
  expressions that pass XLSForm validation but fail at runtime.
- **2.4.7 Calendar** (`#content-32159`)
  Rules for visit scheduling, windowing, conditional visits. Relevant
  when generating event metadata or cross-event references.
- **2.4.8 Using the Content Library** (`#content-9254`)
  Reusable form fragments across studies. Not typically generated by
  this skill but useful context for form_category decisions.
- **2.4.9 Locating Object Identifiers in a Study** (`#content-9701`)
  **Critical for cross-form references.** Documents how Item OIDs,
  Item Group OIDs, Form OIDs, and Event OIDs are named and where to
  find them. The skill's OID naming conventions (F_ prefix, dotted
  itemgroup form) must match what OpenClinica expects here.

## Reading strategy

The OC4 docs site is a single-page app — `web_fetch` typically returns
only the site-wide menu rather than page content. To actually read a
page's body:

1. Fetch the landing URL first and inspect the menu.
2. For the target anchor (e.g. `#content-9204`), fetch the parent
   page URL. The HTML body is client-rendered; check the returned
   markup for server-rendered text before falling back to a browser.
3. If a page body is empty/menu-only, the content is behind JavaScript
   rendering. Note the gap explicitly rather than proceeding with
   assumed content.
4. `references/xlsform-build-rules.md` in this skill directory is the
   distilled, stable reference derived from the OC4 docs + the
   OpenClinica blank form template. When in doubt, that file wins for
   build decisions; fetch the live OC4 docs only when the build rules
   file is silent or ambiguous on a specific question.
